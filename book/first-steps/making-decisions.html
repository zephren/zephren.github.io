Making Decisions

Making Decisions

Up to this point you should be able to write a program that has some variables and can do some math. That is a great starting point, but honestly it's probably not all that interesting yet. As humans, we make many decisions every day, and while not always rational they're most likely founded on some amount of information. Almost all programs make decisions too; however, they are founded entirely on fact. And given the same information, the exact same decision will be made every single time (sadly the same cannot be said for us humans).

What If...

Every decision that a computer makes is based on what is called a statement and whether that statement is true, or whether it is false. Everything that is a value in JS can also be considered a true or a false value. Later in this section I've listed out all the types of values and their truthiness. The most common way for a program to make a decision is to use the if statement:

if (<statement>) {
    // Then...
    <more_code>
}

The structure of an if statement introduces the use of an opening ( { ) and closing ( } ) curly bracket. For the sake of not giving you misinformation, these are not required when used under certain conditions (which we wont get into right now). However, for the purposes of simplicity, think of them as something you must always have when writing an if statement. That way the main structure of your if statement will always be the same. What the curly braces do is allow you to tell JS that the section of code inside them () is only to be executed when the is true. I don't have a good metaphor for if statements other than they seem to line up with how we as humans can perform actions based on objectively making a decision. For example:

  • if the baby is crying, then try to comfort the baby.
  • if the house is on fire, then get out of the freaking house.
  • if the car won't start, then check whether it has any gas.

I'll write these examples out in a moment. But first, here is some code where we are just using the the literal boolean value to show what happens when the statement is true or false:

Something that evaluates to true

if (true) {
    // The code inside the brackets will execute
}

// More code, always executed

Something that evaluates to false

if (false) {
    // The code inside the brackets will NOT execute
}

// More code, always executed

Other examples:

var theBabyIsCrying = true;
if (theBabyIsCrying) {
    // Do something to comfort the baby
}

var theHouseIsOnFire = true;
if (theHouseIsOnFire) {
    // Get out of the freaking house
}

Now is probably a good time to introduce the not operator which is just an exclamation point !. This just takes the opposite of what ever it is next to. So a true value becomes false and a false value becomes true. The not operator always directly precedes whatever it is you want to take the opposite of. So, using the above examples:

var theBabyIsCrying = false;
if (!theBabyIsCrying) {
    // This is good, you are not distracted by the baby
    // Go about your business
}

var theHouseIsOnFire = false;
if (!theHouseIsOnFire) {
    // This is pretty normal
    // Not a whole lot to do here
}

To illustrate the car starting example we need to introduce the else statement. The code contained in the brackets of the else statement will get executed if the if statement evaluates to false.

if (<statement>) {
    // Then...
    <more_code>
} else {
    // Else
    <some_other_code>
}

var theCarStarts = false;
if (theCarStarts) {
    // Yay! you can drive wherever it is you wanted to go
} else {
    // Check to see if it has any gas
}

var theCarStarts = true;
if (!theCarStarts) {
    // Check to see if it has any gas
} else {
    // Yay! you can drive wherever it is you wanted to go
}

The above is the if else statement. From here on I will refer to any combination of if, else, and else if (which you'll see next) generically as an "if statement".

The last thing to add is the else if statement. Let me preface talking about this by giving you some rules for if statements that are pretty simple:

1) There must be one, and only one, "if" in an if statement 2) There can optionally be a single "else" at the end of in an if statement 3) There can be zero or more "else if" statements preceding an "if" statement in an if statement

The else if statement lets you add additional things to check should the original if statement evaluate to false. Since you can have multiple else ifs, think of them as being checked in the order in which you have them listed, whichever one evaluates to true first (if any of them) will get its code between its brackets executed.

I'll give a few examples to hopefully make this more clear:

if (true) {
    // Executes
}

if (true) {
    // Executed
} else {
    // Not executed
}

if (false) {
    // Not executed
} else if (true) {
    // Executed
}

if (false) {
    // Not executed
} else if (false) {
    // Not executed
} else {
    // Executed
}

if (false) {
    // Not executed
} else if (false) {
    // Not executed
} else if (false) {
    // Not executed
} else if (true) {
    // Executed
} else {
    // Not executed
}

Nesting

The last thing I want to give you is the ability to do what is called nesting. This generally refers to putting code that uses brackets (like if statements) inside some other code that uses brackets (again, like if statements). It's much easier shown than explained with words:

if (true) {
    // The following code is executed
    if (false) {
        // Not executed
    } else {
        // Executed
        // We cold keep going with the nesting if we wanted to have more conditions
    }
} else {
    // Not executed
}

var theHouseIsOnFire = true
var theDoorsAreLocked = false
if (theHouseIsOnFire) { // This is true
    // Oh no, we'd better get out
    if (theDoorsAreLocked) { // This is false
        // If this were true, hopefully we could unlock the doors
    } else {
        // Phew, we're able to easily get out of the house
    }
} else {
    // This is just a normal day
}

But Wait, There's More

This should get you going for now. More advanced and concise decision making will be covered in the intermediate topic. If the idea of not knowing the best way of doing something from the start concerns you, then I can offer this piece of advice:

One of the best parts about programming is that if you have done something one way, and then learn a better way to do it, there is nothing stopping you from changing it.

Depending on what you've done, it may be more or less complicated to change it in the future, but that in and of it self is also part of the leaning process. Over time you will get better at writing code than can be updated, rewritten, and maintained.
